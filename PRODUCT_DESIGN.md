# 产品设计文档 - 顶级主厨（TopChef）

## 1. 产品概述

### 1.1 产品名称
顶级主厨（TopChef）

### 1.2 产品定义
一个微信小程序应用，帮助用户记录和管理自己会做的菜肴，包括菜肴信息、制作步骤、食材清单等功能。支持用户与朋友分享菜谱、评论交流、相互关注，形成一个社交化的烹饪知识社区，方便用户随时查阅和分享烹饪经验。

### 1.3 产品目标
- 提供便捷的菜肴记录和管理工具
- 帮助用户管理和组织自己的烹饪知识库
- 支持菜肴搜索和分类查看
- 建立社交分享平台，促进用户之间的菜谱交流
- 提升用户的烹饪体验和效率，增强社区互动

### 1.3.1 版本策略与边界（云端同步优先）
- **当前版本定位（MVP）**：以“云端同步 + 轻社区互动”为核心，确保用户跨设备可用、可分享、可互动。
- **MVP 必做（本期交付）**
  - 微信登录与基础资料
  - 菜肴 CRUD（含食材、步骤、图片）
  - 菜肴浏览/搜索/筛选/排序（服务端分页）
  - 收藏
  - 评论与评分（每用户每菜最多一次，可编辑/删除）
  - 分享（好友/群/二维码）与基础统计
  - 关注与用户主页
  - 隐私可见性（仅自己/仅关注者/公开）与评论开关
  - 内容安全与举报闭环（最小可用）
- **MVP 明确不做（推迟）**
  - 个性化推荐算法、排行榜
  - 多人实时协作点菜（同桌多人同时勾选、冲突合并）；**单人**「聚餐菜单」从菜谱拼装属于 **M3**，与下文分阶段一致
  - 营养计算、成本估算的精确模型
  - 视频教程、直播/商业化链路
- **M3/M4 规划补充（与本文 2.1.8～2.1.10 对齐）**
  - **聚餐点菜（单人）**：用「菜单 + 菜单项」承载，从菜谱加入、调份量/备注、可复制菜单；后续再接购物清单
  - **饮食日记**：独立数据（吃了什么），与「计划菜单」分离；先做按日记录 + 可选关联菜谱，不做营养分析
- **本地能力定位**：仅做“缓存与弱网体验”，不承诺完全离线可编辑；离线期间的编辑走队列，恢复网络后自动同步并提示冲突处理（见技术方案）。

### 1.4 目标用户
- 年龄：18-50岁
- 特征：喜欢烹饪或学习烹饪的用户，对菜谱分享和交流有兴趣
- 需求：需要一个个人菜肴管理工具，同时希望与他人分享和学习烹饪知识

### 1.5 核心价值主张
- **个人化管理**：随时记录和整理自己的拿手菜
- **社交互动**：与朋友分享菜谱，获得反馈和建议
- **知识积累**：浏览其他用户的菜谱，学习新的烹饪技能
- **便利性**：随时随地通过微信小程序访问和查看

## 2. 产品功能需求

### 2.1 核心功能模块

#### 2.1.1 菜肴管理
- **添加菜肴**
  - 支持输入菜名、描述、难度级别、烹饪时间
  - 支持上传菜肴图片
  - 支持设置菜肴标签（如快手菜、家常菜、宴客菜等）
  
- **菜肴详情**
  - 显示基本信息（菜名、图片、难度、时间）
  - 展示食材列表（支持数量和单位）
  - 展示制作步骤（支持文字和图片）
  - 显示标签和收藏状态
  
- **编辑菜肴**
  - 支持修改菜肴信息
  - 支持修改食材和步骤
  
- **删除菜肴**
  - 支持删除菜肴（确认删除）

#### 2.1.2 菜肴浏览
- **菜肴列表**
  - 列表展示所有菜肴（卡片样式）
  - **微信小程序 · 发现首页**：以**搜索**为主入口，**不展示**难度/标签/排序等外露筛选条（避免信息噪音）；服务端仍支持 `difficulty` / `tag` / `sort` 等 query，供其它页面、H5 或后续独立筛选入口复用；首页列表默认按**最新发布**（`sort=latest`）分页拉取。
  - **长列表性能**：发现首页菜谱网格采用**虚拟滚动**（按行裁剪 DOM，触底继续分页加载），减少节点数量、提升滑动流畅度。
  - 支持搜索菜肴（与列表接口 `search` 参数一致）
  - 列表分页加载（`page` / `limit`，与后端 `meta.pagination` 一致）
  - 能力保留（服务端与产品层）：按难度、时间排序；按标签筛选 — 可在「分类查看」、个人菜谱列表或其它产品中定义的前台入口承接，不要求挤在发现首页首屏。
  
- **分类查看**
  - 按标签分类展示
  - 按难度分类展示

#### 2.1.3 收藏功能
- 支持收藏喜欢的菜肴
- 单独的收藏列表页面
- 从详情页快速收藏/取消收藏

#### 2.1.4 评论和评分（新增）
- **菜肴评分**
  - 支持 1-5 星评分
  - 显示菜肴平均评分
  - 显示评分分布图表
  
- **菜肴评论**
  - 支持对菜肴添加评论
  - 显示评论列表（最新/热门排序）
  - 支持删除自己的评论
  - 支持评论通知提醒

#### 2.1.5 菜肴分享（新增）
- **分享菜肴**
  - 支持分享给微信朋友
  - 支持分享给微信朋友圈
  - 支持分享菜肴二维码
  - 分享包含菜肴基本信息和链接

- **分享统计**
  - 显示菜肴被分享的次数
  - 显示菜肴的浏览量

#### 2.1.6 个人中心
- 显示用户基本信息（昵称、头像）
- 显示菜肴统计（总数、按难度分布等）
- 显示最近添加的菜肴
- 显示粉丝/关注数统计
- 支持设置和关于

#### 2.1.7 用户互动（新增）
- **用户主页**
  - 显示用户发布的菜肴列表
  - 显示用户基本信息和粉丝数
  - 支持关注/取消关注用户
  
- **关注功能**
  - 支持关注其他用户
  - 支持取消关注
  - 显示粉丝列表和关注列表

- **账户系统**
  - 微信登录/授权
  - 用户基本信息完善
  - 账户设置和隐私设置

#### 2.1.8 菜单与聚餐点菜（分阶段）
> **产品叙事**：「计划吃什么」用菜单；聚餐场景下，把大家想吃的菜从**菜谱库**加入同一份菜单即可。**不要**与「饮食日记」（实际吃了什么）混在同一数据模型中。

- **一期（M3-A：菜单闭环，单人）**
  - **创建菜单**：多个菜单并存（如：周末聚餐、工作日便当）；名称、描述、标签（含场景标签如「聚餐」）
  - **从菜谱加入菜单**：在菜谱详情或菜单编辑页将菜加入当前菜单；支持份量（或人数备注）、行内备注（如「少辣」）
  - **菜单项顺序**：支持排序（拖拽或上移/下移，实现细节以开发为准）
  - **菜单管理**：列表、编辑元信息、删除、**复制菜单**快速新建（适合「上周聚餐复制一份再改」）
  - **权限与可见性（一期）**：默认**仅创建者**可见与编辑；不在一期做「多人同时编辑同一菜单」
  - **可选轻量汇总**：展示菜单内菜品总烹饪时间、难度分布等（**不做**精确成本与营养汇总）

- **可选演进（未承诺排期；与实现是否采纳无关）**  
  - **整单保存心智**：菜单元信息（名称、描述、标签等）与菜单内菜品列表在**概念上同属一张菜单**。当前实现为 **菜单元信息一次保存 + 菜单项多次请求**（加菜、改份量、排序、移除各走接口），以降低首版复杂度。  
  - **后续可评审**：是否提供 **单一保存入口**（一次提交元信息 + 全量或约定的菜单项快照），使编辑页可用一个「保存」完成整单；技术草案见 `TECHNICAL_DESIGN.md` 菜单接口「可选演进：整单保存」。**落地前仍以分接口为权威实现。**

- **二期（M3-B：购物清单）**
  - 从菜单**一键生成**购物清单：合并相同食材（同单位数量相加）、按品类分组
  - 清单项勾选、手改数量/备注、跨端同步（与技术文档 `shopping_lists` 一致）

- **三期（推迟，原「点菜系统」中的智能化能力）**
  - 「今天吃什么」随机推荐、按标签/食材推荐、营养搭配建议、费用估算等——**单独迭代**，避免阻塞 M3 核心闭环

- **多人协作编辑菜单（分两阶段排期；与 M3-A「仅创建者」不矛盾——协作在后续里程碑开启）**
  - **阶段一 — 弱协作（计划优先于实时方案）**  
    - 多名**经授权**用户可编辑**同一份**菜单（邀请/成员列表、角色如 owner / 可编辑 / 只读等产品细则待评审）。  
    - **同步模型**：以服务端为准；冲突时 **提示并拉取最新**（与技术侧 `ifMatchVersion` + 409 一致），**不要求**双方同时在线。  
    - **可选 — 认领行**：编辑某道菜单项前可「认领」该行（租约过期自动释放），减少多人同时改同一行的摩擦。  
    - **明确不做（本阶段）**：WebSocket 实时推送、多人光标、复杂 OT。
  - **阶段二 — 实时协同（在阶段一之后单独评审排期）**  
    - 中心化 **操作指令** + **WebSocket**（或等价长连接）、广播、断线重连与版本补偿；冲突策略（OT 或简化合并）专项设计。  
    - **演进关系**：阶段一落地时，后端宜保持 **按菜单项原子操作** 的写入语义，便于阶段二将同类操作搬到实时通道（见 `TECHNICAL_DESIGN.md`）。
  - **其它能力（可与上列组合、独立里程碑）**：只读分享链接、「各自提交想吃的菜」提议池、版本历史与审计等——**不阻塞**阶段一最小闭环。

#### 2.1.9 饮食日记（分阶段）
> **定位**：记录**已经吃过的菜**，偏习惯与回顾；字段保持极简，降低记录压力。

- **一期（M4：日记 MVP）**
  - **按日查看**：日历或日期选择器进入「某日吃了什么」列表
  - **新增一条记录**：可选 **关联本账号可见范围内的菜谱**；若无对应菜谱，允许 **自由文本菜名**（如外食、外卖）
  - **可选餐次**：早餐/午餐/晚餐/加餐/其他（可跳过，默认「全天」）
  - **简短备注**：可选（如「聚餐」「食堂」）
  - **编辑与删除**：仅本人；云端同步、多设备一致

- **二期（可选增强）**
  - 简单统计（如本周吃过次数最多的菜）、从日记一键「加入下次聚餐菜单」
  - **不做**：精确营养、热量、医疗级饮食建议

### 2.2 非功能需求

#### 2.2.1 性能
- 列表加载响应时间 < 300ms
- 搜索响应时间 < 500ms
- 评论列表加载时间 < 500ms
- 图片加载采用懒加载
- API 响应时间 < 1s

#### 2.2.2 用户体验
- 支持弱网/离线浏览已加载内容（本地缓存）
- 支持请求失败重试与降级展示（缓存兜底）
- 操作反馈提示（加载、删除等）
- 网络异常友好提示和重试机制
- 云端数据与本地缓存的同步（以云端为准，支持离线编辑队列）

#### 2.2.3 安全性
- 微信登录认证，确保用户身份
- 用户数据加密传输（HTTPS）
- 用户隐私相关数据的访问控制
- 评论、昵称、菜名、描述与图片的内容安全校验（必做）

#### 2.2.4 兼容性
- 支持 iOS 和 Android 微信
- 适配不同屏幕尺寸
- 支持微信相关权限（相册、位置等）

#### 2.2.5 可用性
- API 可用性 > 99.5%
- 支持多设备数据同步
- 支持数据备份和恢复

## 3. 数据模型

### 3.1 主要数据实体

#### 用户（User）（新增）
```
{
  id: string,              // 唯一标识（用户ID）
  wechatOpenid: string,    // 微信OpenID
  wechatUnionid: string,   // 微信UnionID
  nickname: string,        // 用户昵称
  avatar: string,          // 用户头像URL
  phone: string,           // 手机号（可选）
  email: string,           // 邮箱（可选）
  status: string,          // 用户状态（active/banned）
  createdAt: timestamp,    // 注册时间
  updatedAt: timestamp     // 更新时间
}
```

#### 菜肴（Dish）
```
{
  id: string,              // 唯一标识
  userId: string,          // 创建者ID
  name: string,            // 菜名
  description: string,     // 描述
  difficulty: string,      // 难度（简单/中等/困难）
  cookingTime: number,     // 烹饪时间（分钟）
  servings: number,        // 份量
  image: string,           // 菜肴图片URL
  tags: string[],          // 标签列表
  ingredients: Ingredient[],  // 食材列表
  steps: CookingStep[],    // 制作步骤
  rating: number,          // 平均评分（1-5）
  ratingCount: number,     // 评分数量
  isFavorite: boolean,     // 当前用户是否收藏（前端）
  viewCount: number,       // 浏览次数（新增）
  likeCount: number,       // 点赞次数（新增）
  visibility: string,      // 可见性（private/followers/public）（新增）
  commentsEnabled: boolean,// 是否允许评论（新增）
  createdAt: timestamp,    // 创建时间
  updatedAt: timestamp     // 更新时间
}
```

#### 评论（Comment）（新增）
```
{
  id: string,
  dishId: string,          // 所属菜肴ID
  userId: string,          // 评论者ID
  userName: string,        // 评论者昵称
  userAvatar: string,      // 评论者头像
  content: string,         // 评论内容
  rating: number,          // 评分（1-5星）
  createdAt: timestamp,    // 评论时间
  updatedAt: timestamp     // 更新时间
}
```

#### 点赞（Like）（新增）
```
{
  id: string,
  dishId: string,
  userId: string,
  createdAt: timestamp
}
```

#### 分享（Share）（新增）
```
{
  id: string,
  dishId: string,          // 被分享的菜肴ID
  userId: string,          // 分享者ID
  shareType: string,       // 分享类型（wechat_friend/wechat_group/qrcode）
  createdAt: timestamp     // 分享时间
}
```

#### 收藏（Favorite）（新增）
```
{
  id: string,
  userId: string,          // 用户ID
  dishId: string,          // 菜肴ID
  createdAt: timestamp    // 收藏时间
}
```

#### 关注（Follow）（新增）
```
{
  id: string,
  userId: string,          // 关注者ID
  followerId: string,      // 被关注者ID
  createdAt: timestamp    // 关注时间
}
```

#### 菜单（Menu）（新增）
```
{
  id: string,              // 唯一标识
  userId: string,          // 创建者ID
  name: string,            // 菜单名称
  description: string,     // 菜单描述
  tags: string[],          // 菜单标签（如：周末聚餐、家常菜等；标签可表达场景，不必单独枚举 type）
  coverImage: string,      // 菜单封面图片URL
  dishes: {                // 菜单项（实现上对应 menu_items；同一道菜是否允许多行见技术文档约束）
    dishId: string,        // 菜肴ID
    servings: number,      // 份量（或倍率，与 UI 文案一致即可）
    notes: string          // 备注信息（聚餐点菜：口味、人数等）
  }[],
  isPublic: boolean,       // 是否公开分享（协作/只读分享为后续能力）
  shareCount: number,      // 被分享的次数
  createdAt: timestamp,    // 创建时间
  updatedAt: timestamp,  // 更新时间
  version: number          // 乐观锁版本（跨端同步，与后端一致）
}
```

#### 饮食记录（MealLogEntry）（新增）
> 与 `Menu` 分离：**菜单 = 计划**，**饮食记录 = 事实**。

```
{
  id: string,
  userId: string,
  eatenDate: string,       // 日历日 YYYY-MM-DD（按用户选择日期落库，不做复杂时区策略时即用户本地日）
  mealSlot: string | null, // breakfast | lunch | dinner | snack | other | null（null 表示未区分餐次）
  dishId: string | null,   // 可选；关联菜谱时需校验当前用户对该菜的可见性
  title: string,           // 展示用菜名：有关联菜谱时可冗余快照，无则必填自由文本
  notes: string | null,    // 可选备注
  createdAt: timestamp,
  updatedAt: timestamp,
  version: number          // 乐观锁版本（编辑/同步）
}
```

#### 购物清单（ShoppingList）（新增）
```
{
  id: string,              // 唯一标识
  userId: string,          // 用户ID
  menuId: string,          // 关联的菜单ID
  items: {                 // 食材项
    id: string,
    name: string,          // 食材名称
    quantity: number,      // 数量
    unit: string,          // 单位
    category: string,      // 分类（蔬菜、肉类、调料等）
    checked: boolean,      // 是否已购买
    notes: string          // 购买备注
  }[],
  estimatedCost: number,   // 预计费用（一期可手工或留空；精确估算为后续能力）
  createdAt: timestamp,    // 创建时间
  updatedAt: timestamp,    // 更新时间
  version: number          // 乐观锁版本（勾选/改量跨端同步）
}
```

#### 食材（Ingredient）
```
{
```

## 6. 业务规则

### 6.1 数据规则
- 菜名必填，长度 1-50 字符
- 难度级别必选（简单/中等/困难）
- 烹饪时间必填，范围 1-300 分钟
- 至少需要一个食材
- 至少需要一个制作步骤
- 最多支持 5 个标签
- 用户昵称长度 2-20 字符
- 评论内容长度 1-500 字符
- 评分范围 1-5 星

### 6.2 操作规则
- 删除菜肴需要二次确认
- 菜肴修改实时保存
- 搜索支持模糊匹配菜名和标签
- 同一用户同一菜肴只能评论一次（可编辑或删除）
- 用户可修改或删除自己的评论
- 关注功能不可关注自己
- 分享成功后分享数+1，浏览数据实时更新
- 点赞同一用户同一菜肴只能点赞一次（可取消）
- 浏览量统计去重策略（MVP）：同一用户对同一菜肴在 30 分钟内多次进入仅计 1 次浏览（服务端口径）

### 6.3 权限规则
- 用户必须登录后才能添加菜肴、评论、分享、收藏
- 用户只能编辑/删除自己的菜肴和评论
- 用户可查看任何公开的菜肴和评论
- 用户可屏蔽/举报不当评论（可选）

### 6.4 隐私规则
- 用户默认菜肴为私密（仅自己可见），可选择分享给朋友或公开
- 用户可设置隐私等级：仅自己可见、仅关注者可见、公开
- 用户可禁用评论功能

### 6.5 内容安全与治理（新增）
- **发布校验**：菜名/描述/步骤/评论文本进行敏感词检测；图片进行涉黄涉政识别（调用云服务或微信内容安全能力）。
- **用户治理**：支持对评论/菜谱进行举报；被举报内容进入待审核队列（MVP 可先做“人工/后台处理 + 前台隐藏”）。
- **处罚策略**：对多次违规用户限制发帖/评论，严重者封禁账号（与 `status` 字段对应）。

### 6.5 通知规则（扩展）
- 用户发布新菜肴时，通知粉丝
- 菜肴被评论时，通知原创作者
- 用户被关注时，发送通知提醒

### 6.6 菜单与聚餐点菜规则（新增）
- 每个用户最多创建 50 个菜单（数值可随运营调整）
- 菜单名长度 1-30 字符
- 菜单最少包含 1 道菜，最多 20 道菜（聚餐场景同上限制，超出引导拆成两份菜单）
- 菜肴份量范围：0.5-10 份（或等价倍率，与后端 `DECIMAL` 一致）
- **同一菜单内同一道菜**：一期允许 **仅一条菜单项**（与当前库表 `UNIQUE(menu_id, dish_id)` 一致）；若需「同一菜冷盘+热吃两行」用备注区分或后续放宽约束
- 菜单可以基于已有菜单复制创建，快速生成新菜单
- 菜单分享（若开放）：权限模型单独定义；**协作编辑**不在 M3-A 范围
- 购物清单自动合并相同食材，相同单位的数量相加（M3-B）
- 购物清单默认云端保存（支持跨设备同步）；同时可在本地缓存最近一次清单用于弱网访问

### 6.7 饮食日记规则（新增）
- 仅登录用户可写；用户只能查看/编辑/删除**本人**记录
- **eatenDate**：必选；允许补记过去日期，**不建议**预填未来日期（前端可警告）
- **内容约束**：`dishId` 与自由文本**至少一种**能表达菜名；关联 `dishId` 时用户须对该菜可见（不可见则禁止选择或保存失败）
- 单条备注长度上限建议 200 字符；自由文本菜名 1-80 字符
- 单日记录条数软上限（如 50 条）防止误操作刷屏
